# Internationalization (i18n) Guide

This document describes how the internationalization system works in the frontend application.

## Overview

The application uses a custom React Context-based i18n solution that supports multiple languages. Currently supported languages:

- English (en) - Default
- Bosnian (ba)

The system is designed to be easily extensible for additional languages.

## Architecture

### Files Structure

```
frontend/
├── locales/
│   ├── index.ts          # Language config and exports
│   ├── en.json           # English translations
│   └── ba.json           # Bosnian translations
├── providers/
│   └── LanguageProvider.tsx   # Language context provider
├── hooks/
│   └── useLanguage.ts    # Hook to access language context
└── components/
    └── ui/
        └── LanguageToggle.tsx  # Language selector component
```

### Key Components

1. **LanguageProvider**: Wraps the application and provides language context
2. **useLanguage**: Hook to access translation function and language settings
3. **LanguageToggle**: Dropdown component for switching languages

## Usage

### Basic Usage

```tsx
'use client'

import { useLanguage } from '@/hooks/useLanguage'

export function MyComponent() {
  const { t } = useLanguage()

  return (
    <div>
      <h1>{t('common.signIn')}</h1>
      <p>{t('home.subtitle')}</p>
    </div>
  )
}
```

### With Interpolation

Translation strings can include variables using `{variableName}` syntax:

```json
{
  "orders": {
    "ordersFound": "{count} order(s) found"
  }
}
```

```tsx
const { t } = useLanguage()

// Outputs: "5 order(s) found"
t('orders.ordersFound', { count: 5 })
```

### Accessing Current Language

```tsx
const { language, setLanguage, availableLanguages } = useLanguage()

// language: 'en' | 'ba'
// setLanguage: (lang: Language) => void
// availableLanguages: ['en', 'ba']
```

## Translation Key Structure

Keys are organized hierarchically by feature/section:

```
common.*           - Common UI elements (buttons, labels)
nav.*              - Navigation items
home.*             - Home page content
restaurants.*      - Restaurants page
cart.*             - Shopping cart
orders.*           - Order history
auth.login.*       - Login page
auth.register.*    - Registration page
profile.*          - User profile
payment.*          - Payment methods
guest.*            - Guest user messages
mealModal.*        - Restaurant menu modal
address.*          - Address forms
admin.*            - Admin panel
language.*         - Language selector
```

## Adding a New Language

1. **Create translation file**

   Copy `frontend/locales/en.json` to `frontend/locales/xx.json` (where `xx` is the language code)

2. **Translate all values**

   Edit the new file and translate all string values (keep the keys unchanged)

3. **Register the language**

   Edit `frontend/locales/index.ts`:

   ```typescript
   import en from './en.json'
   import ba from './ba.json'
   import de from './de.json'  // Add import

   export const LANGUAGES = {
     en: { name: 'English', flag: '🇺🇸', translations: en },
     ba: { name: 'Bosanski', flag: '🇧🇦', translations: ba },
     de: { name: 'Deutsch', flag: '🇩🇪', translations: de },  // Add entry
   } as const
   ```

4. **Done!**

   The new language will automatically appear in the LanguageToggle dropdown.

## Adding New Translation Keys

1. Add the key and English text to `frontend/locales/en.json`
2. Add the corresponding translation to `frontend/locales/ba.json` (and any other language files)
3. Use the key in your component with `t('your.new.key')`

### Naming Conventions

- Use camelCase for key names
- Group related keys under a common prefix
- Use descriptive names that indicate the purpose

```json
{
  "section": {
    "featureName": {
      "title": "Title text",
      "description": "Description text",
      "buttonLabel": "Button text"
    }
  }
}
```

## Language Persistence

The selected language is automatically persisted to `localStorage` under the key `preferred-language`. When the user returns to the site, their language preference is restored.

## SSR Considerations

The LanguageProvider includes hydration handling to prevent SSR mismatches:

- Initial server render uses the default language (English)
- After hydration, the stored preference is loaded
- The `document.documentElement.lang` attribute is updated to reflect the current language

## Best Practices

1. **Always use translation keys** instead of hardcoded strings
2. **Keep translations consistent** across all language files
3. **Use interpolation** for dynamic content instead of concatenation
4. **Test both languages** when adding new UI text
5. **Follow the key naming convention** for consistency
